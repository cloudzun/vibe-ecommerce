## 已知问题（建议优先修复）

在继续加功能之前，有两个安全问题值得先处理：

### 🔴 问题一：任何人都能查到任意订单

`GET /api/orders/:id` 没有访问控制。只要知道订单 ID，不需要登录就能看到完整信息——包括购买者姓名、邮箱、收货地址。

这是 OWASP Top 10 第一名：**IDOR（不安全的直接对象引用）**。

**修复思路**：
- 登录用户：验证 token，只允许查询 `user_id = req.user.id` 的订单
- 游客：下单时生成一个临时查询 token，存入 sessionStorage，查询时附带验证

### 🔴 问题二：价格由前端传入，后端不校验

下单时，商品价格由前端 JS 传给后端，后端直接写入数据库。攻击者可以修改请求，把价格改成 0.01。

**修复思路**：后端收到 `item.id` 后，从数据库查真实价格，不信任前端传来的 `item.price`。

## 功能扩展路线图

完成 LAB-14 的同学可以把这里当作自己的「下一个项目」来做。
每个 Phase 都有明确的学习目标和验收标准，可以独立完成。

---

### Phase 7 — 安全加固（推荐第一个做）

**学习目标**：IDOR 防护、服务端数据校验、防御性编程

**任务清单**：
- [ ] 修复订单访问控制（IDOR 漏洞）
- [ ] 后端从数据库查真实价格，不信任前端传入
- [ ] `items` 深度校验（每个 item 的 `id` 必须是整数、`quantity` 必须是正整数）
- [ ] 服务启动时检查 `JWT_SECRET` 是否设置，未设置则拒绝启动

**验收标准**：
- 未登录用户访问他人订单 → 403
- 登录用户访问自己的订单 → 200
- 提交 `price: 0.01` 的订单 → 数据库存的是真实价格
- 未设置 `JWT_SECRET` 启动服务 → 进程退出并打印错误

**参考**：LAB-11 安全加固

---

### Phase 8 — 分页与搜索优化

**学习目标**：分页设计（offset vs cursor）、API 参数规范、数据库查询优化

**任务清单**：
- [ ] 商品列表分页（`?page=1&limit=12`，返回 `total`、`page`、`totalPages`）
- [ ] 订单历史分页
- [ ] 搜索从 `LIKE '%keyword%'` 升级为全文搜索（SQLite FTS5 或 PostgreSQL `tsvector`）
- [ ] 前端商品列表加「加载更多」或分页控件

**验收标准**：
- `GET /api/products?page=2&limit=5` 返回第 6-10 条
- 搜索「hub」能匹配「USB-C Hub」
- 订单历史超过 10 条时分页显示

**参考**：LAB-03 开发新功能

---

### Phase 9 — 文件上传

**学习目标**：multipart/form-data 处理、文件安全校验、存储策略选择

**任务清单**：
- [ ] 用户头像上传（限制：JPG/PNG，最大 2MB）
- [ ] 商品图片上传（管理员功能，替换 Unsplash 外链）
- [ ] 文件类型校验（不能只靠扩展名，要检查 magic bytes）
- [ ] 防路径遍历攻击（文件名 sanitize）
- [ ] 存储策略选择：本地文件系统 vs 云存储（OSS/S3）

**验收标准**：
- 上传 `.exe` 文件 → 400 拒绝
- 上传超过 2MB 的图片 → 400 拒绝
- 上传成功后商品图片替换为本地 URL

**参考**：LAB-03 + LAB-11

---

### Phase 10 — 管理后台（触发 SDD 的时机）

**学习目标**：RBAC（基于角色的访问控制）、SDD Brownfield 重构

> ⚠️ 完成这个 Phase 后，代码量预计超过 5,000 行。
> 这正是 LAB-14 里 SDD 的触发条件——你会亲身体验「什么时候需要 SDD」。

**任务清单**：
- [ ] 用户角色系统（`admin` / `customer`，存入 users 表）
- [ ] 管理员专属路由（`/admin/*`），非管理员访问 → 403
- [ ] 商品管理：新增、编辑、下架（软删除）
- [ ] 订单管理：查看所有订单、更新状态（待发货/已发货/已完成）
- [ ] 用户管理：查看用户列表、禁用账号
- [ ] 前端管理页面（`#admin`，登录后显示入口）

**验收标准**：
- 普通用户访问 `POST /api/admin/products` → 403
- 管理员可以新增商品，前端立即可见
- 订单状态更新后，用户订单历史页面显示新状态

**参考**：LAB-13 SDD Brownfield（先用 SDD 分析现有代码，再设计管理模块）

---

### Phase 11 — 实时通知

**学习目标**：Server-Sent Events（SSE）、事件驱动架构、长连接管理

**任务清单**：
- [ ] 订单状态变更时，用户收到实时通知（SSE）
- [ ] 管理员收到新订单提醒
- [ ] 前端通知中心（右上角小红点 + 通知列表）
- [ ] 连接断开自动重连

**验收标准**：
- 管理员更新订单状态 → 用户页面实时显示新状态（无需刷新）
- 断网重连后通知恢复

**参考**：LAB-07 并行 Agent 重构（这个 Phase 改动面广，适合并行执行）

---

### Phase 12 — 可观测性

**学习目标**：结构化日志、请求追踪、性能监控

**任务清单**：
- [ ] 结构化日志（JSON 格式，包含 `requestId`、`userId`、`duration`）
- [ ] 每个请求生成唯一 `correlation-id`，贯穿整个调用链
- [ ] 慢请求告警（超过 500ms 打印 WARNING）
- [ ] 内存缓存升级为 Redis（支持多进程、持久化）
- [ ] `/metrics` 端点（请求数、错误率、响应时间 P95）

**验收标准**：
- 每条日志包含 `requestId` 字段
- 同一请求的所有日志有相同的 `requestId`
- Redis 重启后缓存自动重建

**参考**：LAB-10 性能优化

## 工程原则提醒

在继续之前，回顾一下 LAB-14 里的核心理念：

**每次进入新 Phase 前，先写上下文文档，而不是直接写代码。**

```
# 上下文文档 - Phase X

## 项目背景
当前系统状态：技术栈、已有功能、架构特点

## 之前尝试过什么
上一阶段的成果、遇到的问题、遗留的技术债

## 成功与失败的定义
- 成功标准：可量化的目标
- 失败红线：绝对不能破坏的东西

## 谁会受影响
受影响的功能模块、需要回归测试的内容

## 约束条件
技术约束、不能更换的接口、质量要求
```

写好上下文文档的 AI 输出质量，远高于随手写的 Prompt。
